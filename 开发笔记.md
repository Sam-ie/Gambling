## Android 适配

### 1. QDialog::exec() 闪退

**现象**：Android 上一旦触发任何弹窗（设置回合数、失误率对话框），应用立刻崩溃。

**根因**：Android 不支持嵌套事件循环。`QDialog::exec()` 内部调用 `QEventLoop::exec()`，这在桌面端正常工作，但在 Android QPA 中没有对应的事件循环实现，直接崩溃。这个问题和手写还是用 Qt Designer 的 Dialog 无关，只要调用 `exec()` 就会炸。

**方案**：彻底消灭弹窗。把之前弹出对话框才能设置的内容全部改为直接内嵌在页面里的控件。删除了 4 个设置按钮（设置回合数、淘汰、遗传、失误率），替换为直接放在 settingsTab 里的 SpinBox、CheckBox、Slider。

**教训**：跨平台 Qt 开发，`exec()` 系列方法要慎用。能用内嵌控件解决的绝不用弹窗。

### 2. QComboBox 闪退

**现象**：Android 上点击 QComboBox 展开下拉列表时闪退。试过各种方法都无效：设置 `setMaxVisibleItems()`、给 ComboBox 单独设 stylesheet、强制 fusion style，全部失败。

**根因**：这是 Qt 的已知 Bug（QTBUG-127495）。QComboBox 展开下拉列表时会创建一个新的顶层窗口（`Qt::Popup` 类型）。Android QPA 需要为每个顶层窗口分配一个 Java SurfaceView。多个 Surface 叠加 + JNI 线程调度的问题导致在某些设备上崩溃。本质上和 QDialog 的问题是同一类——Android QPA 对多窗口支持不完善。

**方案**：两个 QComboBox 全部替换为内嵌控件：
- 语言选择：两个紧贴的 QPushButton（中/英），spacing=0，左侧按钮取消右边框，做出分段选择器的效果，checkable + 互斥实现 toggle
- 作弊 NPC 选择：改为只读 QLineEdit，默认显示"请在游戏页双击对应NPC"。实际选择操作移到游戏页——双击 NPCCircleWidget 里的 NPC 圆圈，触发 `npcDoubleClicked` 信号，更新 QLineEdit 内容并自动跳转到设置页

**教训**：Android 上不要用任何需要弹出新窗口的控件。QComboBox、QMenu、QDialog、QToolTip 都是坑。

### 3. 黑屏 + 白色文字

**现象**：Android 上欢迎页和人数设置页黑屏，所有文字变成白色。

**根因**：两层问题叠加。

**第一层**：`gambling.ui` 中 `tabWidget` 设了 QSS 样式（`QTabBar::tab` 的背景色/选中态样式）。Android 上只要某个控件调了 `setStyleSheet()`，它和所有子控件都会从原生渲染切换到 QSS 渲染引擎。QSS 在 Android 上是暗色默认值（黑底白字）。所以所有 tab 子页面——人数设置、游戏设置、帮助页——全部黑底白字。

**第二层**：欢迎页是一个裸 QWidget，`autoFillBackground` 默认 false，它直接透出 Android 设备的默认背景色。之前用全局 stylesheet 覆盖了，误把全局 stylesheet 用 `#ifndef Q_OS_ANDROID` 屏蔽之后，欢迎页就黑掉了。游戏页不黑是因为 NPCCircleWidget 是自绘控件。

**方案**（三处联动）：

1. `main.cpp`：Android 分支强制设置亮色 QPalette（Window=white, Text=black, Button=浅灰），覆盖设备暗色主题
2. `gambling.cpp`：Android 下对 `tabWidget` 调用 `setStyleSheet(QString())` 清除 QSS 触发；各 tab 页设 `autoFillBackground(true)`，让它们使用 QPalette 的亮色背景
3. `gambling.cpp`：欢迎页设置 `autoFillBackground(true)`

**教训**：Android 上 `setStyleSheet()` 有副作用——会触发整个控件树的 QSS 渲染引擎切换。如果只需要改某个特定控件的样式，优先考虑单独 setStyleSheet 只给那个控件，或者在程序启动早期就设好全局样式。另外，裸 QWidget 不会自动填充背景，需要显式开启 `autoFillBackground`。

### 4. Tab 标签宽度不一致

**现象**：修复黑屏之后，Android 上四个 tab 标签宽度参差不齐。"游戏"两个字很窄，"人数设置"四个字很宽，看起来非常不协调。

**根因**：黑屏修复时用了 `ui->tabWidget->setStyleSheet(QString())` 清空整个 tabWidget 的 stylesheet，连带把 `gambling.ui` 里写的 `QTabBar::tab { min-width: 80px; padding: 0px 20px; }` 也清掉了。失去这个约束后，Android 原生样式按文字内容自动计算 tab 宽度。

**方案**：分两步。先对 tabWidget 整体清除 stylesheet，然后单独拿到 tabBar 对象，调用 `setExpanding(true)` 让标签均匀分配空间，再单独给 tabBar 设回 `QTabBar::tab` 样式（font-size、height、min-width、padding）。

**教训**：`QTabWidget::setStyleSheet()` 会同时影响 tabBar 和 tab 内容的样式。需要精准控制时，应该分开操作：tabWidget 清空 QSS，tabBar 单独设置。

---

### 5. Checkbox / Slider 样式丢失

**现象**：Android 上修复黑屏之后，QCheckBox 的勾选框变回了系统默认小框（原来有自定义样式），QSlider 的自定义 handle 样式也消失了。

**根因**：修复黑屏时误判了根因。把 `qApp->setStyleSheet()` 用 `#ifndef Q_OS_ANDROID` 包起来，以为全局 stylesheet 导致了黑屏。但实际上**黑屏的真正原因是 tabWidget 自身的 stylesheet**，不是全局的。去掉全局 stylesheet 之后，Android 上 Checkbox::indicator 和 Slider 的自定义样式全部失效，变回系统原生。

**方案**：去掉 `#ifndef` 限制，全局 stylesheet 恢复全平台生效。Android Palette（亮色）保持在 stylesheet 之前设置，这样 QPalette 负责背景色，stylesheets 负责控件样式，各司其职。

**教训**：分析问题时不要把"时间上的关联"当成"因果"。黑屏和全局 stylesheet 是同时出现的，但不代表全局 stylesheet 是原因。修 bug 应该逐个隔离变量测试，而不是一刀切。

---

### 6. 启动白边 + 横屏锁定

**现象**：Android 上状态栏和导航栏两侧有白边，应用没有撑满屏幕。

**方案**：三重横屏锁定：
- `main.cpp`：`#ifdef Q_OS_ANDROID` 下使用 `showMaximized()` 替代 `show()`
- `.pro`：`ANDROID_APP_ORIENTATION = landscape`
- `AndroidManifest.xml`：`screenOrientation="landscape"`

### 7. make install 目标缺失

**现象**：Android 构建时报 `No rule to make target 'install'`。

**根因**：Android Makefile 的 `apk_install_target` 依赖 `make install`，但 `.pro` 中 `target.path` 只对 `unix:!android` 设了值，Android 没有任何 `INSTALLS` 条目，make 自然不会生成 install 目标。

**方案**：在 `android {}` 块中添加：
```qmake
target.path = /libs/$${ANDROID_TARGET_ARCH}
INSTALLS += target
```

## 布局问题

### 8. 英文版 tab 高度溢出

**现象**：窗口高度 468px，中文模式下各页面正常，切到英文后设置页内容超出屏幕底部。

**排查过程**：一开始以为是英文字体/字号导致的——英文文本通常比中文更宽，可能触发了换行。但检查按钮和标签都没有换行。也不是窗口高度的问题（中文够用的高度英文应该也够）。最后细算了一下**设置页的实际内容总高度**：

| 组件 | 高度 |
|------|------|
| 页面上/下 margin | 16px |
| VBoxLayout spacing × 10 处 (原 8px) | 80px |
| 各类行 (minHeight=36) × 9 行 | 324px |
| 分隔线 + 底部 spacer (原 40px) | ~44px |
| **合计** | **≈ 458px** |

而 tabWidget 实际可用内容区 = 468（窗口高） - 32（tabBar） - ~6（边框）= **≈ 430px**。

430 < 458，**本来就塞不下**。中文勉强能显示是因为中文字符紧凑，渲染时 margin 略小；英文文本稍宽，Qt 重新计算布局时就会顶出去。

**方案**：减小间距，不碰字号。
- VBoxLayout spacing：8 → 4（省 40px）
- 底部 spacer：40 → 4（省 36px）
- 总内容区变为 ≈ 382px，远小于 430px 可用空间

**教训**：页面超出屏幕时，第一步应该算一遍内容高度。不一定是文本变宽了，可能是内容本来就超出了可用空间，只是某个语言模式下刚好差一点才暴露出来。

### 9. NumberPicker 不居中

这个问题经历了三轮排查和修复，是整个项目里最折腾的布局问题。

**第一轮**：所有页面的 NumberPicker 靠右。原因是有一段"修复布局"的代码给**每个** NumberPicker 前面都插了 Expanding spacer，把控件推到右边去了。修复方式是限定只对人数设置页（populationTab）插入 spacer。

**第二轮**：用户要求去掉 spacer，改为 `setAlignment(widget, AlignCenter)` 居中。人数设置页的设置生效了，但淘汰行（设置页）的两个 NumberPicker 还是靠右。

排查发现两个根因：
- **淘汰行是嵌套布局**：外层 `layoutEliminationRow` 包含内层 `layoutElimSub`，递归代码只处理了 `item->layout()` 子布局，跳过了嵌套在内层的 NumberPicker
- 修复递归代码后仍然不居中。进一步排查发现 NumberPicker 构造函数里设了 `setMaximumWidth(130)` + `QSizePolicy::Maximum`。Qt 布局管理器给这个 widget 分配的 cell 宽度等于 widget 的自然宽度（约 110px），在这个等宽 cell 里调用 `AlignCenter` 没有任何视觉效果——widget 已经撑满 cell 了，往哪居中？

**方案**：
- `gambling.ui`：`layoutElimSub` 设置 `stretch="0,1,0,1,0"`，Label 不拉伸，NumberPicker 占剩余空间
- `gambling.cpp`：嵌套布局（depth ≥ 1）里的 NumberPicker 改设 `QSizePolicy::Expanding`，使它的 cell 比 widget 宽，`AlignCenter` 才能真正居中

**第三轮**：人数设置页"强化学习者"的 NumberPicker 比其他行靠右。根因是所有标签设了 `QSizePolicy::Maximum` + `minWidth(0)`，5 个字的"强化学习者"和 3 个字的"诚实者"标签宽度不同，导致 NumberPicker 起始位置不一样。

**方案**：人数设置页标签统一 `setMinimumWidth(100)`，`QSizePolicy::Preferred`，所有 NumberPicker 自然对齐。

**教训**：`maximumWidth` + `Maximum` 策略的组合在 stretch 布局里是个坑——widget 不会自动扩展填满 cell，导致居中对齐失效。如果要居中，要么去掉 maxWidth 限制，要么改 SizePolicy 为 Expanding，确保 cell 比 widget 宽。

---

### 10. 设置页标签列不等宽

**现象**：设置页的 NumberPicker 居中之后，发现不同行的 NumberPicker 竖直方向对不齐。根源是各行 Label 文本长度不同（"双方诚实"3字 vs "积分规则"4字 vs "作弊NPC"不对应的长度），每个主标签列宽度各异。

**方案**：遍历设置页所有顶层 HBoxLayout，对位置 0 的主标签和位置 2 的次要标签，统一设 `setMinimumWidth(140)`。但需要排除 `lblElimination`——它是淘汰行的标题而非列标签，不应受此约束。

所有行标签列宽度一致后，NumberPicker 自动对齐。

**教训**：居中只是让单行内的控件位置好看，多行之间的视觉对齐需要统一列宽度。`minWidth` 比 `fixedWidth` 好——它设定了下限但允许超出，更灵活。

## 编译与构建

### 11. app.ico 丢失

**现象**：桌面端 Debug 构建时报 `debug/Gambling_resource_res.o Error 1`。

**根因**：.pro 里有 `RC_ICONS = app.ico`，QMake 会自动生成 .rc 文件引用这个图标，然后 windres 编译 .rc。但 `app.ico` 文件不存在，windres 失败。

**方案**：从 `android/res/mipmap-xxxhdpi/ic_launcher.png` 重新生成 `app.ico`，并在 .pro 里加了一段自动检测：如果 ico 不存在就从 mipmap PNG 重新生成，避免下次再丢。

---

### 12. ui_welcome.h 损坏

**现象**：Android Release 构建时报 `allocation of incomplete type 'Ui::Welcome'`。检查 `gambling.h` 确实有 `#include "ui_welcome.h"`，不是头文件缺失的问题。

**排查**：用 hexdump 看 `ui_welcome.h`——6766 字节全部是 `\0`。UIC 工具生成的中间文件损坏了。

**方案**：删除损坏的文件，Qt 重新生成。

## 国际化

### 13. 语言切换失效

**现象**：在欢迎页切换语言后，界面不更新。

**根因**：`Ui::Welcome welcomeUi` 是 `Gambling` 构造函数里的局部变量。语言切换时调用 `retranslateUi(widget)` 需要访问这个对象，但它已经在构造函数结束时销毁了。

**方案**：`welcomeUi` 改为 `Gambling` 的成员变量（堆上分配），`onLanguageChanged` 中可以正常调用 `m_welcomeUi->retranslateUi(m_welcomePage)`。

**教训**：UIC 生成的 UI 对象如果需要在多个方法之间共享（不只是构造时一次性 setupUi），必须提升为成员变量。

---

### 14. 两个 QTranslator 冲突

**现象**：中文模式下部分界面显示英文，英文模式下欢迎页语言不切换。

**根因**：`main.cpp` 里残留了一个系统 locale 的 QTranslator 安装逻辑。这个 translator 在整个应用生命周期里从未被移除。当 Gambling 内部安装自己的 translator 时，两个 translator 并存，Qt 的翻译查找机制会去先装的 translator 里找，可能返回错误的翻译。而且多 translator 并存时 `retranslateUi` 行为不确定。

**方案**：删除 `main.cpp` 中所有 QTranslator 相关逻辑（QLocale、installTranslator、QTranslator 本身），语言切换完全由 `Gambling::onLanguageChanged` 管理。一个应用只保留一个 translator，简单可靠。

---

### 15. 演化按钮中英文混乱

**现象**：英文模式下演化按钮的状态切换完全失效——点了"Start"之后按钮状态卡住，不再变为"Pause"或"Resume"。

**根因**：`on_btnEvoStart_clicked` 用 `btnEvoStart->text()` 做字符串比较来判断当前状态：
```cpp
if (btnEvoStart->text() == "开始") { ... }
else if (btnEvoStart->text() == "暂停") { ... }
else if (btnEvoStart->text() == "继续") { ... }
```
英文模式下按钮文本是 "Start" / "Pause" / "Resume"，全部匹配失败。而且某些地方直接用 `setText("暂停")` 硬编码中文覆盖翻译。

**方案**：引入 `EvoBtnState` 枚举（EVO_IDLE / EVO_RUNNING / EVO_PAUSED），所有涉及按钮状态的地方改用枚举判断和设置。`setEvoBtnState()` 内部用 `tr()` 获取当前语言的按钮文本。`onLanguageChanged` 中 `retranslateUi` 后重新调用 `setEvoBtnState` 恢复当前状态的正确文本。

**教训**：永远不要用 UI 文本做程序逻辑的判据。文本会随语言变化，逻辑不应该。用枚举或状态变量代替。

## 架构与重构

### 16. God Class 拆分

**背景**：重构前 `GameEngine` 有 619 行，`Gambling` 有 916 行。两个类包揽了几乎所有逻辑：NPC 管理、交互执行、积分计算、淘汰、遗传、自动演化、UI 信号连接。自动演化的状态管理用了 5 个布尔标志位（`m_autoEvoPaused`、`m_autoEvoAdjustingRound`、`m_autoEvoWaitingAdjust`、`m_autoEvoFastMode`、`m_autoEvoSingleRound`），修改一处需要检查所有 32 种可能的标志位组合。

**拆分方案**：

| 拆出模块 | 职责 | 设计模式 |
|----------|------|----------|
| InteractionRunner | 统一出牌→失误→记录→计分→RL 流程 | 消除 3 处重复代码 |
| EliminationManager | 淘汰 + 积分折算 | 独立子系统 |
| AutoEvolutionEngine + 4 状态类 | 自动演化状态机 | 状态模式替代 5 个布尔值 |
| PayoffMatrix | 支付矩阵计算 | 值对象 |
| NPCConfig | NPC 配置 | Builder 替代 11 参数函数 |
| SpinGroupController | Spin 控件组管理 | 消除 5 处 × 11 行重复 |
| HelpTextBuilder | 帮助文本生成 | 中/英模板分离 |
| PresetManager | 预设配置 | 数据驱动 |

结果：GameEngine 从 619 行精简到 ~200 行，Gambling 从 916 行精简到 ~500 行。每个子模块职责明确、可独立测试。

**教训**：大型重构要渐进式推进。先把最独立的模块拆出来（PayoffMatrix、NPCConfig），再处理耦合度高的（InteractionRunner 依赖 GameEngine 的多个内部状态），最后处理最复杂的（AutoEvolutionEngine 涉及状态机），而不是一次性全部打散重写。

## 演化与博弈

### 17. 演化时出现虚线

**现象**：英文模式下启动自动演化时，NPCCircleWidget 上偶尔会画出不应该存在的虚线（玩家对战连线）。

**根因**：`setGameRunning(true)` 启动演化时没有重置 `m_currentOpponentId`。这个变量存的是上一局玩家对战时最后一个对手的 ID。演化启动时 `drawNPCs()` 检查到 `m_currentOpponentId != -1`，就从圆心画了一条到那个 NPC 的连线。

**方案**：`setGameRunning(false)` 被调用时（游戏结束/重置），同时重置 `m_currentOpponentId = -1`、`m_pairHighlightA = -1`、`m_pairHighlightB = -1`。在 `setNPCData()` 中也清除 pair 高亮标志。

**教训**：从一个模式切换到另一个模式时，要清理上一个模式的所有状态残留。演化模式不需要玩家对战连线，这些标记就应该在进入演化前全部清掉。

---

### 18. 演化首次点击左边不更新

**现象**：启动演化后第一帧 NPCCircleWidget 不刷新，看起来像卡住了。

**根因**：演化启动时调用了 `setNPCData()` → `update()` 请求异步重绘。但在第一帧 `update()` 还没执行时，后续的演化 tick 又触发了新的 `update()` 调用，第一次的重绘请求被覆盖了。

**方案**：`setNPCData()` 末尾改用 `repaint()` 强制同步重绘，确保首次数据设置后控件立即刷新。

**教训**：`update()` 是异步的，多个连续的 `update()` 调用会被 Qt 合批处理。如果某个操作之后需要立即看到结果，用 `repaint()` 强制同步绘制。
